FlowSync
GitHub 协作扩展实施指南
简版 · V1.0
文档目标
• 把现有“项目—任务—进度—总结”流程升级为“仓库—Issue—分支—PR—验收—交付”闭环。
• 明确接下来每个阶段要做什么、完成到什么程度，以及如何判断阶段通过。
• 先完成可演示、可运行的最小版本，再逐步增加 AI 代码验收与自动发布。

依据：《FlowSync 小组任务协同管理系统需求规格说明书》V1.0
建议实施周期：4～6 周

1. 改造目标与系统边界
改造后的 FlowSync 不负责存储和版本管理代码，而是作为项目指挥中心，与 GitHub 形成双向协作。
组成
主要职责
不负责
FlowSync
项目、成员、任务、截止时间、验收结果、交付状态
不自行实现 Git 仓库和代码版本管理
GitHub
仓库、Issue、分支、Commit、Pull Request、Actions、Release
不替代 FlowSync 的教学项目管理页面
AI
拆任务、生成验收标准、审查 PR、总结项目
第一版不自动合并主分支、不代替负责人最终确认

目标业务流程
最终闭环
• 负责人创建项目并连接 GitHub。
• AI 拆解任务，负责人确认后创建仓库、Issue 与任务分支。
• 成员在自己的任务分支提交代码并发起 Pull Request。
• GitHub Actions 自动测试，AI 根据任务验收标准审查修改。
• 负责人确认合并；项目结束后生成 Release 与交付报告。

必须坚持的三条规则
FlowSync 管业务状态，GitHub 管代码事实。
一个任务对应一个 Issue 和一个任务分支，不使用“一个成员一个长期分支”。
AI 提供建议和验收报告，合并主分支与最终发布由负责人确认。
2. 开发前的必要改造（阶段 0）
现有系统采用教学简化认证。GitHub 接入将产生创建仓库、邀请成员、合并代码等高权限操作，因此必须先补齐安全基础。

阶段 0 验收条件
• 用户不能通过修改请求参数冒充其他人。
• 成员无法调用负责人专属接口。
• 所有 GitHub 高风险写操作都有审计记录。

阶段 3：PR、自动测试与负责人验收
成员在 FlowSync 点击“提交验收”，系统创建或关联 Pull Request。
PR 模板自动写入任务编号、实现内容、验收标准和自测说明。
GitHub Actions 执行后端测试、前端构建和代码检查。
FlowSync 接收 check_run / workflow_run Webhook，展示通过或失败。
负责人查看代码差异、测试结果后选择“通过并合并”或“要求修改”。
阶段 4：AI 验收与项目交付
AI 读取任务说明、验收标准、PR diff 和测试结果，输出结构化验收报告。
项目最终验收检查未合并 PR、失败测试、未关闭阻塞 Issue 和缺失文档。
通过后创建 GitHub Release，并生成项目总结、运行说明、贡献统计和已知问题。
推荐开发顺序
• 先完成阶段 0 和阶段 1，再开始任何 GitHub 写操作。
• 课程展示的最小可用版本做到阶段 2 即可形成明显亮点。
• 阶段 3 是完整协作闭环；阶段 4 作为创新能力和后续论文/竞赛扩展。

6. 后端模块建议
建议包结构
• controller：GitHubAuthController、GitHubRepositoryController、GitHubWebhookController、GitHubAcceptanceController。
• service：GitHubAuthService、GitHubRepositoryService、GitHubTaskService、GitHubWebhookService、AcceptanceService。
• integration/github：GitHubApiClient、GitHubTokenProvider、GitHubWebhookVerifier。
• 原则：Controller 不直接调用 GitHub API，所有外部调用集中在 integration 层。

第一阶段验收清单
□ 负责人能绑定和解除 GitHub；成员不能替负责人绑定项目仓库。
□ 项目能绑定一个已有仓库，刷新页面后信息仍然存在。
□ FlowSync 能展示默认分支、最近提交、Issue、PR 和 Actions 状态。
□ Webhook 使用签名校验；重复 delivery_id 不会重复处理。
□ GitHub API 失败时有明确错误提示，数据库状态不会被写坏。
□ 日志中不输出 access token、私钥和模型 API Key。

最终原则
• 不要一开始就做“AI 自动整合全部代码”。先打通授权、仓库绑定、任务发布和状态同步。
• 每完成一个阶段，都要能够独立演示、独立验收，不依赖下一阶段才能运行。
• 正式版本优先使用 GitHub Organization；个人仓库只作为教学演示或早期原型。
